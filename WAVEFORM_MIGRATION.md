# Waveform Feature Migration Guide

## Database Migration

Run this SQL command on your D1 database to add the waveform_data column:

```sql
ALTER TABLE tracks ADD COLUMN waveform_data TEXT;
```

Via Wrangler CLI:
```bash
cd worker
wrangler d1 execute PSC_DB --command="ALTER TABLE tracks ADD COLUMN waveform_data TEXT;"
```

## Worker Deployment

Deploy the updated worker with new endpoints:

```bash
cd worker
npm run deploy
```

This adds:
- `GET /audio/:path` - Serves audio files from R2
- `POST /tracks/:id/waveform` - Saves waveform data to database

## How It Works

### On Upload
1. User uploads audio file → saved to R2 + metadata to D1
2. Client-side analyzer uses Web Audio API to decode and analyze audio
3. Generates two resolutions:
   - **High-res**: 1000 samples for deck waveform
   - **Low-res**: 80 samples for track list preview
4. Waveform data saved to D1 as JSON

### On Playback
1. Track loads with waveform data (if exists)
2. Deck displays frequency-colored waveform (Serato-style)
3. Playhead moves across waveform in sync with audio
4. Click waveform to seek
5. Track list shows preview waveforms (colored if real data exists)

## Existing Tracks

Existing tracks without waveform data will:
- Display placeholder waveforms (seeded deterministic bars)
- Generate real waveforms on next upload of new tracks
- Can be regenerated by re-uploading (duplicate prevention will reject same title+artist+vault)

- [ ] Upload a new track

- [ ] Load track to deck
- [ ] Verify waveform displays with frequency colors (blue/green/orange)
- [ ] Play track
- [ ] Verify playhead moves across waveform
- [ ] Click waveform to seek
- [ ] Check track list preview shows colored waveform
- [ ] Verify existing tracks show placeholder waveforms

## Rollback

If issues occur:
1. No code rollback needed - placeholder waveforms work without data
2. Database column is nullable - safe to leave empty
3. Worker endpoints are additive - no breaking changes
